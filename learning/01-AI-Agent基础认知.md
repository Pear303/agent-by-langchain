# 第1章：AI Agent 基础认知

> **学习目标**：理解 AI Agent 是什么、和普通 AI 有什么区别、Agent 的核心组成部分

---

## 1.1 从普通 AI 到 AI Agent

### 普通 AI 对话

你用过 ChatGPT 或 DeepSeek 聊天吧？典型的交互模式是：

```
你: "中国的首都是什么？"
AI: "中国的首都是北京。"
```

这是 **一问一答** 模式：你提问题，AI 回答。AI 没有记忆（每次对话独立），没有行动能力（只能"说"不能"做"）。

### AI Agent 的飞跃

AI Agent 不是只回答问题，而是**主动完成任务**。

```
你: "帮我查一下这周北京的天气趋势，如果周末天气好就生成一个郊游计划"
AI Agent 会:
  1. 调用天气API → 获取天气预报
  2. 分析数据 → 判断周末天气
  3. 生成计划 → 写入文件保存
  4. 汇报结果 → "已完成！"
```

> **核心区别**：普通 AI 是"问答工具"，AI Agent 是"自主执行者"。

## 1.2 AI Agent 的核心组成要素

一个完整的 AI Agent 由以下四个部分组成：

```
┌──────────────────────────────────────────────┐
│                AI Agent                       │
│                                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │  大脑 LLM │  │ 工具 Tool │  │ 记忆Memory│   │
│  │  (思考决策)│  │ (执行行动) │  │ (记住上下文)│   │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘   │
│       │              │              │         │
│  ┌────┴──────────────┴──────────────┴────┐   │
│  │          规划 Planning                  │   │
│  │     (拆解任务、决定下一步做什么)            │   │
│  └───────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

### ① 大脑 — LLM

大语言模型是 Agent 的"大脑"，负责：
- 理解你的指令
- 分析当前状态
- 决定下一步调用什么工具
- 总结执行结果

> 没有 LLM，Agent 就是一个空壳。

### ② 工具 — Tool

工具是 Agent 的"手脚"，让 Agent 能**影响外部世界**：

| 工具类型 | 举例 | 作用 |
|---------|------|------|
| 文件工具 | `read_file`, `write_file` | 读写文件 |
| 网络工具 | `web_fetch`, `web_search` | 获取网上信息 |
| 执行工具 | `run_command` | 执行 Shell 命令 |
| 搜索工具 | `glob`, `grep` | 在代码库中搜索 |
| 特殊工具 | `update_todos`, `dispatch_subagent` | 管理任务、派遣子代理 |

> Agent 本身什么都不会，但通过工具，它什么都能做。

### ③ 记忆 — Memory

记忆让 Agent **不是"一次性"的**：

| 记忆类型 | 比喻 | 作用 |
|---------|------|------|
| 工作记忆 | 便签纸 | 当前对话的上下文 |
| 情景记忆 | 日记本 | 今天做过什么事的记录 |
| 长期记忆 | 人生履历 | 跨会话的核心知识 |

> 没有记忆的 Agent 每次对话都是"失忆"状态，无法持续完成任务。

### ④ 规划 — Planning

规划是 Agent 的"项目经理"：
- 把大任务拆成小步骤
- 决定步骤的先后顺序
- 实时调整计划

```
原始任务: "搜索香菇资料，做成 HTML 文件"
规划结果:
  1️⃣ 搜索3个以上网页
  2️⃣ 整理总结资料
  3️⃣ 写入 HTML 文件
```

## 1.3 Agent 的工作流程：推理-行动循环

这是 Agent 最核心的工作模式——**ReAct 循环**（Reasoning + Acting）：

```
用户输入: "帮我查天气并生成计划"
       │
       ▼
┌─────────────────────────────┐
│ ① 推理 (Reasoning)           │
│ LLM 思考: "我需要先查天气，    │
│ 再根据结果决定是否生成计划"     │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ ② 行动 (Acting)              │
│ Agent 调用天气查询工具         │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ ③ 观察 (Observation)         │
│ Agent 收到天气查询结果:       │
│ "周六晴，27°C"               │
└──────────┬──────────────────┘
           │
           ▼
     ┌─────┴─────┐
     │ 是否完成?   │
     │ LLM 判断   │
     └─────┬─────┘
           │
     ┌─────┴─────┐
     ▼           ▼
  (继续)      (结束)
  回到①       输出结果
```

这个过程不断重复，直到任务完成或达到最大迭代次数。

### 真实项目中的循环

打开本项目的 `agent/lc_agent.py`，可以看到核心循环：

```python
# AgentExecutor 内部自动执行 ReAct 循环
result = self.executor.invoke({
    "input": user_input,
    "chat_history": self.memory_store.messages,
})
```

你不需要手动写循环——LangChain 的 `AgentExecutor` 已经封装好了这个机制。

## 1.4 Agent 的两大经典架构

### 架构一：单 Agent 模式

```
用户 ──→ Agent（一个LLM + 多个工具）
                 │
     ┌─────┬─────┼─────┬─────┐
     ▼     ▼     ▼     ▼     ▼
  工具1  工具2  工具3  工具4  工具5
```

- **优点**：简单、易于调试
- **缺点**：一个 Agent 做所有事，上下文窗口压力大
- **适用**：简单的单任务场景

### 架构二：多 Agent 模式（本项目的架构）

```
用户 ──→ 主 Agent（指挥官）
              │
     ┌────────┼────────┐
     ▼        ▼        ▼
  子Agent1  子Agent2  子Agent3
  (搜索)    (分析)    (执行)
     │        │        │
     ▼        ▼        ▼
  网页抓取   读文档    写文件
```

- **优点**：分工明确、上下文隔离、可并发执行
- **缺点**：架构复杂、需要设计通信机制
- **适用**：复杂多步骤任务（本项目的目标场景）

本项目实现的就是**多 Agent 模式**——主 Agent 负责规划和协调，子 Agent 负责具体执行。

## 1.5 为什么选择 LangChain？

LangChain 是目前最主流的 **Agent 开发框架**，它提供了：

| 你需要什么 | LangChain 提供 |
|-----------|---------------|
| 统一的 LLM 调用接口 | `ChatOpenAI`、各种 LLM 包装器 |
| 工具定义规范 | `@tool` 装饰器、BaseTool 基类 |
| Agent 运行循环 | `AgentExecutor` 自动 ReAct 循环 |
| 记忆管理 | `BaseChatMessageHistory`、多种 Memory 实现 |
| 提示词管理 | `PromptTemplate`、`ChatPromptTemplate` |
| 回调系统 | `BaseCallbackHandler` 事件钩子 |

**不用 LangChain 会怎样？** 你需要手写：
1. LLM API 调用的消息格式转换（JSON ↔ Python 对象）
2. ReAct 循环（判断→执行→观察→再判断）
3. 工具调用的参数校验和错误处理
4. 对话历史的序列化和反序列化
5. ... 以及很多其他琐碎工作

> LangChain 不解决"能不能做"的问题，它解决"做得快不快、稳不稳定"的问题。
